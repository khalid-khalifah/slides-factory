# v0.4 — Feature 4.1: Brand Theme Inheritance

**Status:** ✅ Complete — Ready for Chief Programmer sign-off

**Branch:** `v0.4-brand-inheritance`

---

## What Was Built

### `_deep_merge(base, override)` — private helper
Recursive dict merge with list-replace semantics: child lists replace parent lists (never append). Child scalars override parent scalars. Nested dicts merge recursively.

### `_load_brand_raw(path, _chain=None)` — private resolver
Reads a brand YAML, resolves `extends` inheritance, and returns the fully merged raw dict (before `BrandTheme` construction).

Key behaviours:
- `extends` accepts a **single string** or a **list of paths** (ordered merge)
- Later `extends` entries override earlier ones; the child's own values override everything
- Resolution is relative to the child's parent directory (`path.parent / rel_path`)
- Circular chains detected via a private `_chain` parameter → raises `SlidesFactoryError`
- Chains deeper than 5 levels → `logger.warning()`
- Missing parent files → standard `FileNotFoundError`

### `load_brand(path)` — updated
Now delegates to `_load_brand_raw()` for the merge, then constructs `BrandTheme` from the merged dict. Backward compatible: YAML files without `extends` load identically to before.

---

## API Surface

No new public API. The `extends` key appears only in brand YAML files. `load_brand()` retains its existing signature.

### Usage

```yaml
# brand.yaml  (child)
extends: base.yaml
name: "Acme Brand"
colors:
  main:
    - color: "#00A3E0"
      contrast: "#FFFFFF"
```

```yaml
# brand.yaml  (multi-inheritance, ordered merge)
extends:
  - slides-factory-basic.yaml
  - acme-colors.yaml
colors:
  main:
    - color: "#FF6600"   # child wins over both parents
      contrast: "#FFFFFF"
```

### Merge Rules

| Key | Strategy |
|-----|----------|
| `extends` | Consumed during resolution, excluded from merged output |
| `colors.main/secondary/basic` | Child **replaces** parent list entirely |
| `fonts.slots` | Child overrides by slot key |
| `layout.elements` | Shallow merge — child overrides by element key |
| `layout.logos` | Deep-merge — child locale entries override parent |
| `page` | Child **replaces** parent entirely (field-level merge not attempted) |
| `logos` | Child overrides by logo key |
| `name`, `default_frame`, `base_pptx`, `lock_frame_shapes` | Child wins if present |

# v0.4 — Feature 4.3: Layout Debug Mode

**Status:** ✅ Complete — Ready for Chief Programmer sign-off

**Branch:** `v0.4-debug-mode`

---

## What Was Built

### `slides_factory/layout/debug.py` — `render_debug_layer()`

A dedicated debug module that renders diagnostic shapes **behind** the normal
slide content (python-pptx creates shapes in order, so debug shapes come
first and content appears on top).

**Rendered diagnostics:**

| Layer | What | Colour | Look |
|-------|------|--------|------|
| Playground boundary | Dashed red rectangle | `#FF0000` | 1pt dash |
| Padding region | Semi-transparent gold fill at playground edges | `#FFD700` | 15% alpha |
| Grid lines | Dashed gray column/row track boundaries | `#888888` | 0.25pt dash |
| Gap regions | Light fill bands between tracks | `#DDDDDD` | 12% alpha |
| Cell boundaries | Thin solid blue per-cell rectangle | `#3366FF` | 0.25pt solid |
| Cell labels | `[row,col]` text in top-left corner | Black | 8pt |

### Integration — `ctx.debug` flag on `RenderContext`

Rather than threading a `debug` parameter through every function in the call
stack, the flag is stored on `RenderContext.debug` (a frozen dataclass field).
`render_layout()` checks `ctx.debug` and calls `render_debug_layer()` before
rendering content. All intermediates — `RenderContext.with_debug()`, engine
`prepare_render(debug=True)`, document API — exist to propagate the flag from
the CLI to the context.

### CLI commands updated with `--debug`

| Command | `--debug` added |
|---------|----------------|
| `slide new` | ✅ — renders grid frame, lines, labels |
| `slide add` | ✅ — renders behind template content |
| `slide add-many` | ✅ — applied to every rendered slide |

### Python API

```python
# All three document functions accept debug=True
result = document.new_grid_slide(prs, app=app, grid="grid-cols-2 grid-rows-2", debug=True)
result = document.add_slide(prs, "simple", data, app=app, debug=True)
results = document.add_slides_from_rows(prs, "simple", rows, app=app, debug=True)
```

# v0.5 — Feature 5.1: Headless PNG Export

**Status:** ❌ Dropped after design review

**Reason:** Pillow cannot faithfully rasterize the wide variety of PPT/X
shapes (gradients, shadows, 3D bevels, charts, SmartArt, custom geometry).
The 4–6 hours of effort would produce an unreliable subset renderer.

**Remaining path:** LibreOffice (`soffice`) remains the supported preview
engine. The existing `pptx_bytes_to_pngs()` in `preview/render.py` continues
unchanged.

# v0.5 — Feature 5.2: Template Composition

**Status:** ✅ Complete — Ready for Chief Programmer sign-off

**Branch:** `v0.5-template-composition`

---

## What Was Built

### `@at(template=...)` — mutually exclusive with `kind=`

```python
@app.template("hero-kpi", grid="grid-cols-1 grid-rows-[1_3] gap-4")
class HeroKpi(Template):
    @at("col-span-1", template="hero-heading")
    def heading(self): ...

    @at("col-span-1", kind="card")
    def metrics(self): ...
```

### Changes made

| File | Change |
|------|--------|
| `slides_factory/templating.py` | `CellDef` gains `template` field + `_validate_cell_def()`; `at()` accepts `template=`; `Template.build()` emits template cells as `CellSpec(template=..., cell_data=...)` |
| `slides_factory/layout_spec.py` | `CellSpec` gains optional `template` + `cell_data` fields; `element` becomes optional |
| `slides_factory/registration.py` | `input_model_from_template()` resolves sub-template input model when `cell.template` is set |
| `slides_factory/layout/render.py` | `render_layout()` detects template cells and renders sub-template with a sub-playground context |
| `tests/test_composition.py` | **New** — 6 composition tests |

### How it works

1. **Registration** — `input_model_from_template()` traverses `@at` cells. When
   `cell.template` is set, it looks up the sub-template via `factory.get_template()`
   and uses its `input_model` as the nested field type. This also catches
   unknown template ids early.

2. **Build** — `Template.build()` emits template cells as `CellSpec(template=...,
   cell_data=...)` instead of `CellSpec(element=...)`. Cell data is dumped as
   a raw dict (no element validation).

3. **Render** — `render_layout()` checks each cell. When `cell.template` is set:
   - Validates the cell data against the sub-template's input model
   - Creates a sub-context with the cell's EMU box as the playground
   - Calls `sub_template.render(slide, validated_data, sub_ctx)`
   - The sub-template builds and renders its internal grid inside the cell box

4. **Nested composition** works recursively — each sub-template can itself
   contain `@at(template=...)` cells.

5. **Circular references** are detected at registration time (when resolving
   the input model), since `factory.get_template()` is called synchronously.

### Constraints

- Sub-template's `default_frame` is ignored — the parent's frame controls the slide
- Placement attributes (e.g., `col-span-2`) size the cell in the parent grid
- The sub-template receives the cell box as its playground and renders independently
