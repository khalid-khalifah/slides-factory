# v0.4–v0.5 Feature Guide

This document covers the features added in the v0.4 and v0.5 development
cycles. It is intended as a user-facing reference — how to *use* each feature
— rather than an implementation log.

---

## Brand Theme Inheritance (`extends`)

A brand YAML file can inherit settings from one or more parent brands and
override only what differs.

**Single parent:**

```yaml
# child-brand.yaml
extends: base-brand.yaml

name: "Acme Corp — Q3 Override"
colors:
  main:
    - color: "#00A3E0"
      contrast: "#FFFFFF"
```

**Multiple parents (ordered merge):**

```yaml
# child-brand.yaml
extends:
  - shared-colors.yaml     # lowest priority
  - acme-defaults.yaml     # medium priority

name: "Acme Corp"
colors:
  main:
    - color: "#FF6600"     # highest priority — wins over both parents
      contrast: "#FFFFFF"
```

Later entries override earlier ones; the child's own values override everything.

### Merge Rules

| Key | Behaviour |
|-----|-----------|
| `colors.main/secondary/basic` | Child **replaces** parent list entirely (no appending) |
| `fonts.slots` | Child overrides by slot key (title, body, footer, …) |
| `layout.elements` | Shallow merge — child overrides by element key |
| `layout.logos` | Deep merge by locale (child `en` replaces parent `en`) |
| `page` | Child **replaces** parent entirely |
| `logos` | Child overrides by logo key |
| `name`, `default_frame`, `base_pptx`, `lock_frame_shapes` | Child wins if present |

### Error Handling

| Situation | Result |
|-----------|--------|
| Circular chain (`a → b → a`) | `SlidesFactoryError` with chain trace |
| Missing parent file | `FileNotFoundError` with resolved path |
| Chain deeper than 5 levels | `logger.warning()` — not an error |

## Layout Debug Mode (`--debug`)

The `--debug` flag renders diagnostic shapes behind your slide content so you
can verify the grid layout visually. No more guessing where cells fall.

```bash
# CLI
your-slides slide new deck.pptx --grid "grid-cols-[2_1] grid-rows-2 gap-4 p-4" --debug
your-slides slide add deck.pptx --template hero --debug
your-slides slide add-many deck.pptx --template kpi --rows data.json --debug

# Python API
document.new_grid_slide(prs, app=app, grid="grid-cols-2", debug=True)
document.add_slide(prs, "hero", data, app=app, debug=True)
document.add_slides_from_rows(prs, "kpi", rows, app=app, debug=True)
```

### What Gets Rendered

| Layer | Description | Colour |
|-------|-------------|--------|
| Playground boundary | Dashed rectangle around the body region | Red |
| Padding region | Semi-transparent fill at playground edges | Gold |
| Grid lines | Dashed lines at column/row track boundaries | Gray |
| Gap regions | Light fill bands between tracks | Light gray |
| Cell boundaries | Thin solid rectangles per cell | Blue |
| Cell labels | `[row,col]` text in top-left corner | Black, 8pt |

All debug shapes are rendered **before** content, so they appear behind it.

## Template Composition (`@at(template=...)`)

A template cell can delegate to another template instead of containing a
single element. This lets you compose common patterns from smaller, reusable
pieces.

### Example

```python
# Sub-template — a single heading cell
@app.template("hero-heading", name="Hero Heading", grid="grid-cols-1 grid-rows-1")
class HeroHeading(Template):
    @at(kind="text")
    def headline(self): ...

# Parent template — one cell is the sub-template, another is a card element
@app.template("hero-kpi", name="Hero + KPIs", grid="grid-cols-1 grid-rows-[1_3] gap-4")
class HeroKpi(Template):
    @at("col-span-1", template="hero-heading")   # ← composes the sub-template
    def heading(self): ...

    @at("col-span-1", kind="card")               # ← regular element cell
    def metrics(self): ...
```

### Input JSON

```json
{
  "heading": {
    "headline": {"block": {"children": [{"runs": [{"text": "Q3 Review"}]}]}}
  },
  "metrics": {
    "title": "Revenue",
    "value": "$1.2M"
  }
}
```

The sub-template's `default_frame` is ignored — the parent's frame controls the
slide. Placement attributes (`col-span-2`, etc.) size the cell in the parent
grid; the sub-template renders its own internal grid inside that cell.

### Constraints

| Rule | Behaviour |
|------|-----------|
| Nested composition | Works recursively (template → sub-template → sub-sub-template) |
| Circular references | Detected at registration time |
| Mixed cells | You can mix `kind=` (elements) and `template=` (sub-templates) in the same template |
| Mutually exclusive | `kind` and `template` cannot both be set on one `@at` |

## Box Geometry Object

The `Box` class replaces ad-hoc `(left, top, width, height)` tuples throughout
the codebase. It is **immutable, hashable**, and backward-compatible with tuple
unpacking.

```python
from slides_factory.geometry import Box

b = Box(left=100, top=200, width=500, height=300)

# Destructure like a tuple
left, top, width, height = b        # 100, 200, 500, 300

# Compare with tuples
assert b == (100, 200, 500, 300)    # True

# Derived properties
b.right          # 600  (left + width)
b.bottom         # 500  (top + height)
b.center_x       # 350.0
b.center_y       # 350.0

# Derivation (all return new Box instances)
b.inset(10, 20)              # Box(110, 220, 480, 260) — shrink from all edges
b.with_margin(top=5, left=10)  # Box(110, 205, 485, 295) — directional margins
```

## Element Sizing Constraints

Elements can declare optional minimum and maximum dimensions in EMU at
registration time. The render engine checks these constraints against the
cell box and **warns** (but renders anyway) when they are violated.

### Registration

```python
@app.element(
    "card",
    props_model=CardProps,
    style_model=CardStyle,
    min_width=Emu(1_270_000),   # ~1 inch
    max_height=Emu(5_080_000),  # ~4 inches
)
def render_card(slide, box, props, style, ctx): ...
```

### Template Min/Max

Templates expose `_min_box()` and `_max_box()` class methods that return the
bounding box implied by their elements' constraints:

```python
min_box = MyTemplate._min_box()  # Box | None — smallest Box that fits all cells
max_box = MyTemplate._max_box()  # Box | None — largest Box before any cell overflows
```

These are used internally during template composition — when a sub-template is
rendered inside a parent cell, the engine checks that the cell box satisfies the
sub-template's size requirements and warns if it does not.

### Behaviour on Violation

| Violation | Result |
|-----------|--------|
| Cell narrower than `min_width` | `logger.warning()` — renders anyway |
| Cell wider than `max_width` | `logger.warning()` — renders anyway |
| Cell shorter than `min_height` | `logger.warning()` — renders anyway |
| Cell taller than `max_height` | `logger.warning()` — renders anyway |
